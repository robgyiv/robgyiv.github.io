avail

Hosted service

Avail is an availability-aware scheduling tool designed to reduce coordination overhead by making schedules easy to publish and consume. Each user gets a canonical, shareable URL that returns both a clean plaintext view (/username) and a structured JSON API (/username.json), so it works equally well in a browser, via curl, or inside automations.

Under the hood, it pairs a Preact + Vite frontend with a Go API and PostgreSQL (via Supabase), with a routing setup that preserves simple public URLs while still supporting authenticated updates and API token workflows.

Open source library

Avail: The Availability Tool I Wanted (So I Open-Sourced It)

I built avail because the workflow I wanted didn’t really exist: a fast, trustworthy way to generate clean availability from my real calendar without pushing people into yet another scheduling tool. Most products try to book the meeting. I just wanted to assist the conversation.

avail is a local-first, privacy-first CLI that reads your calendar, computes free/busy time within your working hours, and outputs human-friendly availability you can paste into email, Slack, or anywhere else. It can also push derived availability only to a hosted API when you want a shareable link — event details never have to leave your machine.

What it does well

  • Generates up-to-date availability from your actual calendar
  • Stays out of the way: terminal-first, clipboard-friendly output
  • Keeps a strong privacy boundary: read-only integrations and derived data sharing
  • Works across multiple input modes: Google Calendar (OAuth), public .ics URLs, or local .ics files

How it works (under the hood)

The project is split into a few clean layers:

  • cmd/avail/ wires up the CLI entry point
  • internal/cli/ implements commands like auth, show, copy, and push
  • internal/calendar/ defines a provider interface and multiple implementations (Google API, public URL, local file)
  • pkg/engine/ is the core availability engine
  • pkg/availability/ holds the core types (Event, TimeBlock, WorkHours)

The core engine is intentionally small and explicit. It:

  1. Normalises the date range to day boundaries
  2. Filters events that overlap each day
  3. Subtracts those events (plus configurable buffers) from working hours
  4. Emits only blocks large enough for your meeting duration
  5. Groups and formats the results for humans

That design keeps the scheduling logic testable and provider-agnostic. Calendar providers just return []Event; the engine does the rest.

Why open source

This tool sits close to sensitive data. Open-sourcing it is a feature, not a vibe. You can inspect the privacy model, run it locally, and adapt it to your own workflow. The availability engine is reusable by design, and the provider boundary makes it straightforward to add new integrations.

If you’ve ever thought “I don’t want a booking page — I just want accurate availability,” avail is for you.
